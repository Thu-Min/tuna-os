What is an Operating System?

An Operating System (OS) is the software that runs your computer.
It starts when you press the power button and stays running the entire time.

Every app you use — your browser, your text editor, your games —
they all depend on the OS to work.

What does an OS actually do?

Manages hardware

It talks to the CPU, RAM, keyboard, screen, and other devices so your apps don't have to.

Runs programs

It loads apps into memory, gives them CPU time, and cleans up when they're done.

Manages memory

It decides which program gets which chunk of RAM, and makes sure programs don't step on each other.

Provides services

Reading files, printing text, getting keyboard input — the OS provides these so every app doesn't have to reinvent them.

Think of it like a building manager

Analogy

Imagine an office building. The building manager (the OS) handles electricity, water, elevators, and security. The tenants (your apps) just show up and do their work — they don't need to know how the plumbing works.

Without the building manager, every tenant would need to generate their own electricity and dig their own well. That's what it would be like to run programs without an OS.

What is a Kernel?

The kernel is the core of the operating system.
It's the very first piece of software that runs after the computer starts up.

It sits between your apps and the hardware, acting as a translator and traffic controller.

The layers of a computer

Your Apps (browser, games, shell)
↓ "Hey kernel, I need something" ↓
Kernel (the brain of the OS)
↓ talks directly to hardware ↓
Hardware (CPU, RAM, keyboard, disk)

Apps can't talk to hardware directly — they must ask the kernel.
This keeps the system safe and organized.

Why can't apps talk to hardware directly?

The problem

If every app could directly control the hardware, one buggy app could crash the entire system, overwrite another app's data, or even damage hardware.

The solution

The CPU has two modes: kernel mode (full power) and user mode (restricted). Only the kernel runs in kernel mode. Apps run in user mode and must ask the kernel for permission to do anything with the hardware.

What we're building: Tuna OS

Language: C

We write everything in C, the same language used for Linux and Windows kernels. We only use assembly (low-level CPU instructions) where absolutely necessary.

Platform: x86_64

This is the type of CPU in most computers and laptops. 64-bit means it can handle large amounts of memory.

Output: serial (text)

Instead of drawing to a screen, we print text to a simple serial port — like a terminal. This keeps things simple so we can focus on OS concepts.

Testing: QEMU emulator

We don't need real hardware. QEMU pretends to be a computer, so we can test our OS safely on our laptop.

Step 1

Booting: Waking Up the Computer

When you press the power button, the CPU doesn't know what to do yet.
It needs someone to load our kernel into memory and start running it.

That "someone" is a bootloader — we use one called GRUB.

The boot process, simplified

  1. You press the power button
  2. The motherboard runs its firmware (BIOS/UEFI) — basic hardware checks
  3. Firmware finds and loads the bootloader (GRUB) from disk
  4. GRUB finds our kernel file and loads it into memory
  5. GRUB jumps to our kernel's entry point — our code starts running!

How does GRUB know our file is a kernel?

We put a special "ID badge" at the start of our kernel — a Multiboot2 header.
It contains a magic number that GRUB recognizes. 


// This is our "ID badge" — GRUB looks for this magic number
.set MB2_MAGIC, 0xE85250D6

.section .multiboot2
mb2_header_start:
    .long MB2_MAGIC        // "I am a valid kernel!"
    .long 0                // architecture: x86
    .long (mb2_header_end - mb2_header_start)   // header size
    .long -(0xE85250D6 + 0 + (mb2_header_end - mb2_header_start)) // checksum
mb2_header_end:

Switching from 32-bit to 64-bit mode

GRUB starts us in 32-bit mode (an older, simpler mode).
But we want 64-bit mode — it can address way more memory and is what modern OSes use.

To switch, we need to:

  1. Set up basic page tables (a map of memory — explained later)
  2. Tell the CPU to enable 64-bit mode by flipping special flags
  3. Turn on paging (the memory mapping system)
  4. Jump into 64-bit code

Our first sign of life!

Once we're in 64-bit mode, we can run C code.
The first thing we do? Print a message to prove it worked. 


void kernel_main(uint64_t multiboot2_addr) {
    serial_init();   // set up the serial port for output
    serial_write("kernel: hello from tuna os!\n");
    // We're alive!
}
        

$ make run
kernel: hello from tuna os!
Milestone

We went from a completely blank machine to running our own C code. Everything from here builds on this foundation.

Step 2

Teaching the CPU the Rules

The CPU needs two important "lookup tables" to function properly.
Think of them as rule books that tell the CPU how to behave.

Rule Book 1: Memory Segments (GDT)

The Global Descriptor Table defines memory regions and their permissions.
Most importantly, it defines two privilege levels:

Ring 0 — Kernel Mode

Full access to everything. Only our kernel code runs here. It can talk to hardware, manage memory, and do anything.

Ring 3 — User Mode

Restricted. User programs (like the shell) run here. They can't touch hardware directly — they must ask the kernel.


gdt[1] = make_descriptor(0x9A, 0xA0);  // Kernel code (full access)
gdt[2] = make_descriptor(0x92, 0xC0);  // Kernel data (full access)
gdt[3] = make_descriptor(0xF2, 0xC0);  // User data   (restricted)
gdt[4] = make_descriptor(0xFA, 0xA0);  // User code   (restricted)

Rule Book 2: Error & Event Handlers (IDT)

The Interrupt Descriptor Table tells the CPU:
"When something happens, here's what function to call."

Analogy

It's like an emergency contact list. Divide by zero? Call handler #0. Keyboard pressed? Call handler #33. A program wants the kernel's help? Call handler #128.

Some examples:

  • Error #0 — Division by zero (a math mistake in code)
  • Error #14 — Page fault (a program tried to access memory it shouldn't)
  • Event #32 — Timer tick (the clock fires every 10ms)
  • Event #128 — System call (a program is asking the kernel for help)
Step 3

Reacting to Events

Hardware devices (keyboard, timer, disk) need a way to get the CPU's attention.
They do this by sending interrupts — "hey, something happened!"

What is an interrupt?

Analogy

Imagine you're reading a book (the CPU running code). Someone taps your shoulder (an interrupt). You put a bookmark in your page (save your place), deal with the interruption, then go back to reading from your bookmark (resume the code).

The CPU does exactly this: it saves its current state, runs the interrupt handler, then resumes where it left off.

The interrupt controller (traffic cop)

Multiple devices can send interrupts. The PIC (Programmable Interrupt Controller) acts as a traffic cop — it receives signals from devices and forwards them to the CPU one at a time. 


void pic_init(void) {
    // Tell the PIC to start its setup sequence
    outb(0x20, 0x11);  outb(0xA0, 0x11);

    // Assign interrupt numbers to hardware devices
    outb(0x21, 0x20);  // Timer, keyboard, etc. → start at #32
    outb(0xA1, 0x28);  // Disk, mouse, etc.     → start at #40

    // For now, only listen to the timer (ignore other devices)
    outb(0x21, 0xFE);  // Enable only IRQ 0 (timer)
    outb(0xA1, 0xFF);  // Disable all others
}

The timer: our OS heartbeat

We set up a hardware timer that fires 100 times per second.
Every time it fires, our kernel gets a chance to do housekeeping — like deciding which program should run next. 


#define TARGET_HZ  100   // 100 ticks per second = every 10 milliseconds

void pit_init(void) {
    uint16_t divisor = 1193182 / TARGET_HZ;  // hardware math
    outb(0x43, 0x36);                         // configure the timer
    outb(0x40, divisor & 0xFF);               // set the speed
    outb(0x40, (divisor >> 8) & 0xFF);        // (low byte, then high byte)
}

// This runs automatically 100 times per second:
void pit_tick(void) {
    ticks++;
    schedule();  // pick the next program to run!
}
Step 4

Managing Memory

Your computer has RAM — fast storage that programs use while they're running.
The kernel needs to hand out chunks of RAM to programs and take it back when they're done.

Three layers of memory management

Each layer builds on the one below it, like floors of a building:

Layer 3: Heap (kmalloc / kfree)
Like malloc in C — give me N bytes of memory
↓ uses ↓
Layer 2: Virtual Memory (address translation)
Gives each program its own "view" of memory
↓ uses ↓
Layer 1: Physical Memory (page allocator)
Tracks which 4 KB chunks of actual RAM are free or used

Layer 1: Physical memory — tracking real RAM

We split all of RAM into 4 KB pages (small, equal-sized chunks).
A simple bitmap tracks which pages are free (0) or used (1). 


RAM split into 4 KB pages:

Page:    [0] [1] [2] [3] [4] [5] [6] [7] ...
Bitmap:   1   1   1   0   0   1   0   0  ...
          ↑           ↑
       used         free — can be given to a program
        

// Ask for a free page of RAM
uint64_t page = pmm_alloc_frame();   // returns address of a free 4KB block

// Give it back when done
pmm_free_frame(page);

Layer 2: Virtual memory — every program gets its own "world"

Analogy

Imagine two people each think they live at "123 Main Street" — but their mail goes to completely different physical houses. That's virtual memory: every program thinks it has the same addresses, but the kernel maps them to different physical RAM behind the scenes.

This is how programs are isolated from each other — one program can't accidentally (or intentionally) read another program's memory. 


// Map virtual address 0x400000 to physical page 0x1A000
// with read+write permissions for user programs
vmm_map_page(0x400000, 0x1A000, VMM_FLAG_PRESENT | VMM_FLAG_WRITE | VMM_FLAG_USER);

Layer 3: The heap — easy memory allocation

With the lower layers in place, we can now offer a simple interface:
kmalloc (give me some memory) and kfree (I'm done with it). 


// Allocate memory — just like malloc() in normal C programs
uint64_t *a = kmalloc(64);    // give me 64 bytes
uint64_t *b = kmalloc(128);   // give me 128 bytes

*a = 0xDEADBEEF;   // use the memory
*b = 0xCAFEBABE;

kfree(a);           // done with a — return it
kfree(b);           // done with b — return it

Under the hood: kmalloc asks the virtual memory layer for pages,
which asks the physical memory layer for actual RAM.

Step 5

Running Multiple Programs

A computer feels like it runs many programs at once, but a single CPU core can only run one thing at a time.

The trick? Switch between them so fast that it looks simultaneous.

What is a process (task)?

Each running program is a task. The kernel keeps a record card for each one: 


struct task {
    uint64_t     id;       // unique ID (task #1, task #2, etc.)
    uint64_t     rsp;      // where this task's CPU state is saved
    uint32_t     state;    // READY, RUNNING, or DEAD
    uint8_t     *stack;    // this task's own private stack memory
    struct task *next;     // pointer to the next task (linked list)
};
Analogy

It's like a teacher managing students. Each student (task) has their own desk (stack) and worksheet (state). The teacher goes around the room giving each student a few seconds of attention before moving to the next one.

Context switching: the magic trick

When the timer fires (100 times/second), the kernel can pause the current task and resume another one. This is called a context switch. 


What happens during a context switch:

1. Timer fires → interrupt handler runs
2. Save current task's registers (its "brain state") onto its stack
3. Pick the next task to run (round-robin: just go to the next one in line)
4. Load the next task's saved registers from its stack
5. Return → CPU is now running the new task seamlessly

This happens so fast (~microseconds) that you'd never notice the switching.

The context switch in code

This is one of the few places we must use assembly — we need to directly manipulate CPU registers. 


switch_to:
    // Save the current task's state
    push %rbp            // save all important registers
    push %rbx
    push %r12
    push %r13
    push %r14
    push %r15
    mov  %rsp, (%rdi)   // save the stack pointer

    // Load the next task's state
    mov  %rsi, %rsp     // switch to the other task's stack
    pop  %r15            // restore its registers
    pop  %r14
    pop  %r13
    pop  %r12
    pop  %rbx
    pop  %rbp
    ret                  // jump back into the other task's code
Step 6

Letting Programs Talk to the Kernel

User programs run in restricted mode — they can't access hardware directly.
So how does a program print text or read input?
It asks the kernel using a system call.

What is a system call?

Analogy

A system call is like ringing the service bell at a hotel front desk. You (the program) can't go behind the counter yourself, but you can ring the bell and ask the staff (the kernel) to do something for you — like "please print this text" or "please read a character from the keyboard".

Our kernel supports these system calls:

  • read — get a character from keyboard input
  • write — print text to the screen
  • listfiles — show all files
  • exec — run another program
  • exit — end the current program

How a system call works


User program                              Kernel
───────────                              ──────

1. "I want to print 'hello'"
2. Put syscall number in a register
3. Put arguments in registers
4. Execute special instruction ──────►  5. CPU switches to kernel mode
   (int 0x80)                           6. Kernel reads the registers
                                        7. Kernel does the work (prints text)
8. CPU returns to user mode  ◄──────    8. Kernel puts result in register
9. Program continues

System calls in code

What the program does:


// "Ring the bell" with int 0x80
static int64_t syscall2(
    uint64_t nr,    // which service?
    uint64_t a1,    // argument 1
    uint64_t a2     // argument 2
) {
    int64_t ret;
    __asm__ volatile (
        "int $0x80"
        : "=a"(ret)
        : "a"(nr), "D"(a1), "S"(a2)
    );
    return ret;
}

// Example: print "hello"
syscall2(SYS_WRITE, "hello", 5);

What the kernel does:


// Handle the "bell ring"
void syscall_dispatch(
    struct interrupt_frame *f
) {
    switch (f->rax) {
    case SYS_READ:
        // read from keyboard
        break;
    case SYS_WRITE:
        // print text to serial
        break;
    case SYS_EXEC:
        // load & run a program
        break;
    case SYS_EXIT:
        // end the program
        break;
    }
}

Entering user mode for the first time

An interesting chicken-and-egg problem: to "return" from kernel to user mode, the CPU uses a special instruction called iretq (interrupt return).

But user code hasn't run yet! So we fake an interrupt return — we set up the stack as if an interrupt had happened, then "return" from it. 


We set up the stack to look like this:

 "Where to return to"  →  entry point of the user program
 "What mode to use"    →  user mode (ring 3)
 "Interrupts enabled?" →  yes
 "User's stack"        →  a fresh stack we allocated for the program

Then we call iretq and the CPU "returns" into the user program!
Step 7

Files and Loading Programs

We need two things: a file system to store programs,
and a program loader to read and run them.

A simple file system in RAM

Instead of reading from a hard drive (which is complex), we keep files in memory.
The user programs are baked into the kernel binary at build time. 


// At boot time, register the embedded programs as "files"
ramfs_init();
ramfs_create("hello.elf", hello_data, hello_size);
ramfs_create("shell.elf", shell_data, shell_size);

// Now we can look them up by name:
const struct ramfs_file *f = ramfs_find("hello.elf");
// f->data has the file contents, f->size has the length
Analogy

It's like packing a lunch box. Instead of stopping at a restaurant (reading from disk), we packed the food (programs) into our bag (kernel) before we left home (at build time).

Loading programs: the ELF format

Programs are stored in a format called ELF (Executable and Linkable Format).
It's the standard format on Linux and Unix systems.

An ELF file is like a shipping package with a label: 


ELF file structure:
┌─────────────────────────────────────────────┐
│ Header                                       │
│   - Magic bytes: 0x7f 'E' 'L' 'F'          │  ← "I'm an ELF file"
│   - Architecture: x86_64                     │  ← "I'm for this CPU"
│   - Entry point: 0x400000                    │  ← "Start running here"
├─────────────────────────────────────────────┤
│ Segment 1: code (the instructions)           │  ← Load at address X
│ Segment 2: data (variables, strings)         │  ← Load at address Y
└─────────────────────────────────────────────┘

Our loader reads the header, copies segments into memory,
then jumps to the entry point.
Step 8

The Shell: Putting It All Together

The shell is a command-line interface — it's the first thing
you see when the OS finishes booting.

It's a user program, not part of the kernel itself.
It uses everything we've built so far.

What the shell does


The shell loop:

1. Print "tuna> " prompt
2. Wait for you to type a command
3. Read what you typed, character by character
4. When you press Enter:
   - "ls"   → ask the kernel to list files
   - "exit" → ask the kernel to shut down
   - anything else → try to run it as a program name
5. Go back to step 1

The shell code

Notice: no libraries, no printf, no scanf. Just raw system calls. 


void _start(void) {
    char line[128];

    for (;;) {
        write_str("tuna> ");       // print the prompt (uses SYS_WRITE)

        // Read a line character by character (uses SYS_READ)
        uint64_t pos = 0;
        for (;;) {
            char c = read_char();
            if (c == '\n') break;  // Enter pressed
            line[pos++] = c;
            write_char(c);         // echo the character back
        }
        line[pos] = '\0';

        // Handle the command
        if (str_eq(line, "ls"))
            syscall0(SYS_LISTFILES);       // list files
        else if (str_eq(line, "exit"))
            syscall0(SYS_EXIT);            // quit
        else
            syscall2(SYS_EXEC, line, ...); // try to run as program
    }
}

Everything connects: the layers at work

When you type exec hello.elf in the shell, here's what happens across all our layers: 


You type "exec hello.elf" and press Enter

Shell (user mode, ring 3)
  └─ syscall(SYS_EXEC, "hello.elf")    ← asks the kernel via int 0x80

Kernel (kernel mode, ring 0)
  ├─ ramfs_find("hello.elf")           ← finds the file in our RAM filesystem
  ├─ elf_load(file_data, file_size)    ← parses the ELF, copies code into memory
  │    ├─ pmm_alloc_frame()            ← gets physical RAM pages
  │    └─ vmm_map_page()              ← maps them into the program's address space
  └─ jump to entry point (iretq)      ← switches to user mode, runs hello

Hello program (user mode, ring 3)
  └─ syscall(SYS_WRITE, "Hello from ELF!")  ← prints its message
  └─ syscall(SYS_EXIT)                      ← exits, kernel returns to shell

The complete picture

Every step we built, in order. Each one depends on the ones before it.

   │
   ▼

   ▼

   ▼

   ├── syscall_init()      ← system call handler
   ├── ramfs_init()        ← in-memory file system
   └── launch shell        ← jump to user mode, run the shell!

Key takeaways

  • An OS is not magic — it's software that manages hardware for your programs
  • The kernel is the core — it's the first thing that runs and it stays running forever
  • Everything is built in layers — each layer depends on the ones below it
  • Programs can't touch hardware directly — they ask the kernel via system calls
  • Multitasking is an illusion — the CPU switches tasks so fast you don't notice
  • Memory management gives each program its own private world
  • You can build all of this with C code and a bit of assembly
