# Concerns

## Boot GDT Duplication

`boot.S` contains a hardcoded bootstrap GDT (`gdt64`) that is conceptually duplicated by the proper GDT in `gdt.c`. Both define null, code, and data segments with similar but not identical encodings. As the GDT grows (e.g., TSS, user-mode segments), maintaining two separate GDT definitions increases the risk of inconsistency.

## Port I/O Helpers Not Shared

`serial.c` defines `outb`/`inb` as `static inline` functions. Future hardware drivers (PIT, PIC, keyboard) will need the same primitives, which would lead to repeated definitions across modules.

## Fixed-Size Stack (16 KiB)

The kernel stack is allocated as a 16 KiB BSS region in `boot.S`. There is no guard page or stack overflow detection. Deep call chains or large stack-allocated buffers could silently corrupt adjacent memory.

## No IST (Interrupt Stack Table) Usage

All IDT entries have `ist = 0`, meaning exceptions reuse the current stack. A double fault or stack overflow during exception handling would cause undefined behavior rather than a clean crash.

## Build Artifacts Checked In

`kernel/build/` and `kernel/iso/` contain compiled object files, the ELF binary, and the ISO image. These are regenerated by `make` and would normally be gitignored.

## Identity Map Covers 1 GiB

The page tables in `boot.S` identity-map the first 1 GiB using 2 MiB pages. This is sufficient for early boot but will need to be replaced with a proper virtual memory manager before higher-half kernel or user-space support.

## No Hardware Interrupt Support Yet

Only CPU exception vectors (0–31) are installed. There is no PIC/APIC initialization and no IRQ remapping, so hardware interrupts (timer, keyboard) cannot be handled. Interrupts remain disabled (`cli` from boot, never `sti`).
